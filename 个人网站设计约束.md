# 个人学习仪表盘 · 架构设计文档

> **一句话概括**
> 一个跑在**本地**的 Go 程序,每天采集 git / Obsidian / Zotero 的学习痕迹,经**手动勾选确认**后推送到个人网站仓库;网站是**纯静态展示层**(仪表盘、热力图、每日日报);其中的 agent 负责生成「**今天相比昨天做了什么**」的对比日报。

---

## 1. 项目目标

- **采集本地三源**:git 今日提交、Obsidian 今日改动笔记、Zotero 今日阅读记录。
- **手动确认上传**:采集后由我勾选哪些内容上传,不是全自动。
- **数据可视化**:上传的数据同时被「仪表盘」和「热力图」读取,呈现工作情况。
- **长期跟踪**:对上传的 Obsidian / Zotero 资料做类似 git 的长期跟进(**不需要版本管理**,只要能回溯时间线)。
- **Agent 功能**:点击按钮后,调用 API key,生成「今天相比昨天的变化」对比日报,推送到网页对应位置。

---

## 2. 核心架构判断:两半式

公网上的静态网页**碰不到本地文件**(浏览器安全沙箱),因此系统必然分成两半:

| | 本地半 · 我的电脑 | 网页半 · 公网静态站 |
|---|---|---|
| 职责 | 采集、勾选确认、agent 总结、推送 | 只做展示 |
| 能否读本地文件 | 能 | 不能 |
| 持有 API key | 是(安全) | 否(绝不放入) |
| 形态 | Go 程序 + localhost 页面 | 静态站(GitHub Pages / Vercel 等) |

**「上传」这个动作 = 本地程序把勾选的数据 `commit & push` 到网站仓库,站点自动重建。**
网页永远不直接读本地,它读的是被推上去的那份数据快照。

---

## 3. 组件与数据流

**本地半**

```
git 今日提交 ─┐
Obsidian 今日笔记 ─┼─►  daily-agent(Go)  ─►  localhost 勾选页  ─►  push 到网站仓库
Zotero 今日阅读 ─┘        采集 · 编排           勾选确认 · 点“总结”
```

- 三个**读取器(tools)**:各自负责读一个源今天的数据。
- **daily-agent 主程序**:调度读取器、跑 agent 循环、执行 push。
- **localhost 勾选页**:唯一有真实逻辑的前端。列出今天采集到的条目 + 勾选框,提供「总结」「上传」两个按钮。**确认和总结都发生在这里**(因为数据和 API key 都在本地最安全)。

**网页半**

```
网站仓库(数据快照) ─►  仪表盘 / 热力图 / 每日日报页
```

- 纯静态,构建时读仓库里的数据文件生成页面。

---

## 4. 数据契约(仓库布局)

一切的地基。数据契约一旦定死,采集端和前端就能**并行开工**。

```
site-repo/
  data/
    days/2025-07-02.json     # 当天原始数据：commits / notes / papers / report
    index.json               # 汇总：喂给热力图和仪表盘
  reports/2025-07-02.md      # 日报正文（人可读）
```

**两个关键设计**

- **一份数据,两个消费者**:同一批按天的 JSON,既生成当天日报页,又被热力图 / 仪表盘聚合读取。
- **git 历史 = 长期跟踪**:每天上传就是一次 commit,git 天然给出「哪天读了什么、记了什么」的时间线。因此**不需要自己造版本管理**——push 到仓库这个动作本身就满足了「类似 git 的长期跟进」。

## 5. Agent 设计

### 5.1 关键澄清:要的是「diff」,不是「总结」

「今天相比昨天做了什么」的核心是**对比**,不是**归纳**。

- **总结**:无状态。喂今天的文档,输出一段话。
- **对比**:有状态,天然需要「昨天」作为参照。

### 5.2 记忆已经存在,第一版不需要 memory 系统

参照物(昨天)不是要从向量库「回忆」出来的,而是**从磁盘读的**——就是每天 push 上去的 `reports/*.md`。

因此第一版只需要:

```
prompt  +  今天的 3 份文档  +  昨天的 1 份日报
```

**不要在入门阶段就上向量数据库 / RAG / 记忆框架**——对「相邻两天对比」这个任务是过度设计,只会陷进基础设施,学不到 agent 核心。

Agent 的输入大致长这样:

```
[系统提示] 你是我的学习记录助手。对比昨天和今天，指出今天新增/推进/中断了什么。
[昨日日报] <reports/2025-07-01.md 的内容>
[今日原料]
  git:      <今天的提交列表>
  obsidian: <今天改动的笔记摘要>
  zotero:   <今天读的文献>
[要求] 输出：今天相比昨天的变化（新开的、继续推进的、搁置的）。
```

### 5.3 什么时候才真的需要 memory 管理

判断线:**当「参照物」长到塞不进 context,或需要跨越很多天做检索时**,才需要 memory。本质是「context 放不下时的应对策略」——要么压缩,要么检索。

### 5.4 循序渐进的四个阶梯

| 阶段 | 任务 | 记忆策略 |
|---|---|---|
| 1 | 单文档总结(hello-agents 级) | 无记忆 |
| 2 | **相邻对比(当前目标)** | **短记忆 = 读昨天 1 个文件** |
| 3 | 周期回顾(如本周至今) | 压缩记忆:滚动摘要 |
| 4 | 长期问答(如「上次卡在 CUDA 哪) | 检索记忆:向量库 |

**现在动手做第 2 步**,但按这个架子设计,后面往上加不用推倒重来。

### 5.5 用接口预留升级位(Go)

用一个接口把「怎么拿到参照物」抽象出来,今天用最笨的实现,以后换压缩 / 检索时,agent 主循环一行都不用改:

```go
// 参照物提供者：今天先用最简单的实现，以后可无痛升级
type MemoryProvider interface {
    Context(ctx context.Context, today time.Time) (string, error)
}

// 第 2 步：只读昨天的日报
type YesterdayReport struct{ dir string }

func (y YesterdayReport) Context(_ context.Context, today time.Time) (string, error) {
    path := filepath.Join(y.dir, today.AddDate(0, 0, -1).Format("2006-01-02")+".md")
    b, err := os.ReadFile(path)
    if errors.Is(err, os.ErrNotExist) {
        return "（昨天没有记录）", nil   // 第一天要能优雅处理
    }
    return string(b), err
}

// 以后升级：换成 RollingSummary{} 或 VectorRetrieval{}，主循环不动
```

Agent 主循环:`memory.Context()` 拿参照物 → 拼进 prompt → 调 LLM → 出 diff → 写今天的日报。

## 6. 技术选型

### 6.1 主力语言:Go

- **契合系统形状**:本机、每天定时、并发地从三源采集——正是 goroutine + errgroup 的主场;编译成单个静态二进制,cron / launchd 一挂就跑,没有虚拟环境和依赖的麻烦。
- **更利于学 agent**:Go 没有重型 agent 框架,得**自己手写 tool-use 循环**(工具定义、调用、结果回填、多轮、停止条件),框架会把这些机制藏起来,自己写一遍才真懂 agent 怎么转。
- **符合目标**:正在学 Go,这是个难度与收益配比合适的真实项目。

> 注:平时主用 Python,但基于「喜欢有挑战、能学到东西的项目」这一偏好,选择直接上 Go。若某天想快速验证某个 agent 想法,也可临时用 Python 打样。

### 6.2 前端:AI 代写,自己不碰

- **公网站**:纯展示,读 JSON / markdown 生成页面。
- **localhost 勾选页**:虽然有逻辑,但很简单,同样由 AI 代写。
- 全程不需要自己学前端。

### 6.3 不使用的

- **C / C++**:本项目是 I/O 密集的胶水 + agent 编排,不是计算密集,它们没有发挥空间。诚实判断,不硬塞。

### 6.4 安全红线

- **API key 只存在于本地**,绝不进入公网静态页。
- 未来的「转发到 GitHub」按钮:公网页不能直接持有 token,须走 **serverless 函数**(token 存其环境变量),按钮 → 调函数 → 函数用 GitHub API 执行动作。

## 7. 分阶段实施路线

**v1 · 打通最小闭环(无记忆 → 短记忆)**

1. 三个读取器的 stub(先返回假数据)。
2. FastAPI 之外——本项目用 Go,起一个 localhost 勾选页(net/http 即可)。
3. `MemoryProvider` 接口 + `YesterdayReport` 实现(读昨天日报)。
4. tool-use 主循环(工具定义、调用、结果回填),LLM 调用留成一个待填函数。
5. push 逻辑:写 `days/*.json` + `reports/*.md`,commit & push。
6. 让整条链路先空跑起来。

**v2 · 逐个填真**

- 把三个读取器接上真实数据源:
  - git:`os/exec` 调 `git log --since=midnight`,或用 go-git。
  - Obsidian:遍历 vault,挑今天 mtime 改过的 `.md`。
  - Zotero:只读打开本地 `zotero.sqlite`(避开锁)。
- 前端读真实数据,做出仪表盘 / 热力图 / 日报页。

**后续增强**

- 压缩记忆(阶段 3):滚动摘要,日报里带「本周至今」。
- 检索记忆(阶段 4):真需要时再上向量库。
- 「转发到 GitHub」按钮:serverless + GitHub API(gist / 提交文件 / 触发 workflow)。

---

## 8. 关键决策速查

- **两半式架构**:本地程序负责读取 + 确认 + 总结 + 推送;公网站只展示。
- **确认与总结都在本地**:数据和 API key 都在本地,最安全。
- **上传 = git push**:git 历史即长期跟踪,无需自造版本管理。
- **一份数据,两个消费者**:按天 JSON 同时喂日报页和可视化。
- **Agent 要 diff 不要 summary**:第一版只需 prompt + 今天3文档 + 昨天1日报,不上 memory 系统,但用 `MemoryProvider` 接口预留升级位。
- **不过早上向量库**:记忆是「context 放不下」时才引入的应对策略。
- **主力 Go,前端 AI 代写,C/C++ 不用,API key 不进公网**。

---

### 附:一个容易踩的 Go 小坑

日期格式化不用 `YYYY-MM-DD`,而是参考时间 `2006-01-02`(完整参考:`01/02 03:04:05 PM 2006`)。第一次会觉得莫名其妙,记住即可。
